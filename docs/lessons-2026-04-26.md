# Lessons learned — 2026-04-26 long-term-memory build

A working note from the session that built the **long-term memory** feature
on top of `feat/artifacts`, plus everything that had to be fixed along the
way. Written so future-me (or anyone touching this code cold) gets the
"why", not just the "what".

Branch: `feat/long-term-memory` — tip `c3b1ecf` at time of writing.

---

## Architecture in one paragraph

`cc-server.py` is the bridge. It holds **one `Worker` per chat session**
in `state.workers: dict[str, Worker]`. Each Worker owns a long-lived
`claude` subprocess started with `--input-format stream-json
--output-format stream-json --resume <sid>`. The Worker never dies on
chat-switch — switching just changes which `sid` the next `send` is
tagged with. **Clients (browser tab, Telegram bot, `cc-talk.py` CLI) are
all peers**: they connect to `ws://127.0.0.1:18793/ws`, send
`{type:"send", id:<sid>, text:"…"}`, and all of them receive every
broadcast (`assistant_start` / `assistant_delta` / `assistant_end` /
`turn_done` / `save_started` / `save_done`). Multi-tab and out-of-band
control work for free.

## Big lessons

### 1. Inherited env destroys spawned-claude auth

When you `nohup python3 cc-server.py …` from inside an active Claude
Code session, the spawned `claude` subprocess inherits **every**
`CLAUDE_CODE_*` / `ANTHROPIC_*` env var the parent had. That includes
`CLAUDE_CODE_OAUTH_TOKEN`, `ANTHROPIC_BASE_URL`, etc. — which override
the user's local `claude login` config and produce **401
authentication_error** against their account. The standalone CLI works
fine; only the bridged spawn 401s, which is super misleading.

Fix is in `scrubbed_env()`: explicit strip of known auth keys plus
prefix-strip of `CLAUDE_CODE_*` and `CLAUDE_INTERNAL_*`. **Always
launch the bridge from a clean env**:

```bash
env -i HOME="$HOME" PATH="$PATH" SHELL="$SHELL" USER="$USER" \
  CC_SERVER_PORT=18793 CC_DATA_DIR=$HOME/claude-ui/_data \
  CC_CWD_ROOT=$HOME/claude-ui \
  CC_MEMORY_FILE=$HOME/claude-ui/long-term-memory.md \
  CC_TELEGRAM_BOT_TOKEN=$(cat ~/.openclaw/telegram.token) \
  CC_TELEGRAM_ALLOWED_USERS=6370291936 \
  nohup python3 server/cc-server.py >$HOME/claude-ui/bridge.log 2>&1 &
```

Commit: `83916cd`.

### 2. Smooth streaming = decouple network from render

The old streaming code did `bubble.innerHTML = md(buffer)` every 360 ms.
Every tick blew away parsed structures (lists, code blocks, tables) and
rebuilt them — that's the visible "jump" the user complained about.

Fix is the rAF drainer in `ui/index.html`:

- **One global text buffer** (`bubble.dataset.raw`); deltas append; no
  paint here.
- **One rAF loop** drains chars at adaptive cps (base 90, up to 30× on
  big buffers via `STREAM_MAX_SPEEDUP`). Fractional char budget
  accumulator stops integer-floor from capping below the configured
  cps.
- **Append-only DOM**: `.md-frozen` (parsed once, never touched again,
  appended to via `insertAdjacentHTML`) for completed paragraphs;
  `.md-live` for the in-flight tail. Boundary detection: the largest
  `\n\n` not inside an open ` ``` ` fence.
- **On chat-switch resume**: `appendDelta` dumps the buffered tail at
  once; the rAF catches up at the high adaptive cps so the user sees
  ~2 s of typewriter, not 12 s.

Commits: `bde2b8e`, `b0807d1`.

### 3. Per-Worker state, not global state

Every time we kept the silent-save / saving-UI flag global
(`state.silent_turn = True`, `summarizing = true`), it broke under
multi-chat use:
- saving chat A would put chat B's button in busy state
- chat A's silent_turn flag would gate broadcasts for chat B
- "saving" status would show in the wrong header

Lesson: **as soon as the architecture is multi-worker, every flag
needs to live on the worker, not the State.** Same on the UI side —
`savingChats: Set<sid>` instead of a single `summarizing` boolean.
`isCurrentSaving()` returns true only when the focused chat is in
the set; other chats stay fully usable in the background.

Commit: `a129549`.

### 4. Silent same-process injection (the save mechanism)

Goal: write two files (per-chat MEMORY.md + index entry append) in
the chat's own claude — so the reflection has full conversational
context — without polluting the chat transcript.

```python
# server-side
w.silent_turn = True
w.silent_meta = {'sessionId': sid, 'mdPath': str(md_path),
                 'memoryPath': str(MEMORY_FILE)}
payload = {'type':'user',
           'message':{'role':'user','content':[{'type':'text','text': prompt}]}}
w.proc.stdin.write((json.dumps(payload)+'\n').encode())
```

The reader gates on `w.silent_turn`: skips
`assistant_start`/`assistant_delta`/`assistant_end` broadcasts and
skips `append_message` so the synthetic turn never lands in
`chat.json`. On `result`, fires `save_done` instead of `turn_done`,
clears the flag, marks `lastSavedAt` on the session.

Tool calls (the two `Write`s) execute normally — the gating is
broadcast/persistence-only, not tool-call-level. Claude can write
files all day during a silent turn.

Commit: `ee7a2d9`.

### 5. Parallel tool calls in one assistant message

A turn has one assistant message; that message can contain **multiple
tool_use blocks** which the runtime executes concurrently. The save
prompt explicitly demands this:

> *"Issue BOTH Write tool calls in a SINGLE assistant message — in
> PARALLEL, not sequentially. No narration, no thinking out loud, no
> preamble. Just the two Write blocks, in one message."*

Outcome: the per-chat MEMORY.md write and the index-append write fire
together rather than `Write 1 → wait → Write 2`. Save latency
dropped substantially.

Commit: `b0807d1`.

### 6. Refresh-not-append for the index

Every save used to append a fresh entry to `long-term-memory.md`
without removing the previous one. After 5 saves of the same chat
the index had 5 stale entries.

Fix: server-side regex pre-removal keyed on the entry's
`**Session**: <sid>` line. The bridge clears the prior entry, then
claude appends the new one. Each entry now requires that key:

```
## 2026-04-26 16-30-00 — chat title
- **Session**: <sid>     ← REQUIRED: refresh key
- **Path**: /abs/path/MEMORY.md
- **Triggers**: …
- **Skill**: …
```

The per-chat memory file is also fresh-not-append: stable filename
`<chat-cwd>/MEMORY.md` overwritten on every save. No more
`PROGRESS-<ts>.md` sprawl.

Commit: `16a0036`.

### 7. Persona secrecy: inline content, ban file-read narration

Original persona prompt told claude to "Read PERSONA.md and
INSTRUCTIONS.md at session start". Claude faithfully announced this
on its first turn:

> *"I'll read the persona and instructions files first, then help
> you with the weather forecast."*

Fix is twofold:
1. **Inline the persona/instructions content directly** into the
   system prompt — no on-disk files for claude to "Read".
2. **Strict secrecy clause** that explicitly bans phrases:
   `"I'll read X"`, `"let me check Y"`, `"first I need to look at"`,
   `"persona"`, `"instructions"`, `"system prompt"`, `"PERSONA.md"`,
   `"chat.json"`, `"the cwd"`, etc.
3. Defensive: `cwd / 'PERSONA.md'` and `INSTRUCTIONS.md` are
   `unlink()`ed at every spawn so even if a stale copy existed,
   `Read` would fail.

But — for **memory**, attribution is *desirable*. The user wants to
see continuity working. So the memory preamble explicitly
**encourages** disclosure ("Based on what I've got saved from when
you set up X…") while still banning performative phrasing ("first
rodeo!", "memory directory doesn't exist").

Commits: `68e224d`, `6aa3344`.

### 8. Save UX iterations (the path to "right")

| Version | What | Why it failed |
|---|---|---|
| V1 | Spawn separate opus summarizer subprocess + modal showing file path | Slow; modal leaked internals ("Saved silently in dialog") |
| V2 | Silent same-process inject + "Saving — silent reflection in flight" banner | Banner string leaked the implementation |
| V3 | No banner, no modal — just save button busy class | Global flag → busy class flashed on every chat |
| V4 | Per-chat `savingChats` set + full-pane overlay | Overlay was inside `#scrollarea` → scrolled out of view |
| V5 | Per-chat `savingChats` + inline pill above composer + ✓ Saved toast | Final |

The pill is `position: relative` inside `.composer-wrap` so it always
floats above the input regardless of message-list scroll position.
The toast is `position: fixed` at top center, fades in/out via CSS
animation over 1.8 s.

Commits: `a129549`, `b0807d1`.

### 9. Visible-tree file layout

User strongly preferred **everything under one visible tree**, no
`~/.openclaw`, no `/tmp`. Final layout:

```
~/claude-ui/
├── long-term-memory.md          # cross-chat index
├── bridge.log                    # server log
├── <chat-folder>/                # per-chat working dir
│    ├── chat.json                # message mirror
│    ├── MEMORY.md                # per-chat memory (overwritten on save)
│    └── uploads/                 # files claude wrote
└── _data/                        # bookkeeping (visible, no leading dot)
     ├── cc-sessions/<sid>.json   # full session state
     ├── cc-uploads/<sid>/        # legacy upload location
     ├── cc-telegram.json         # TG chat-id ↔ session map
     └── personas.json            # persona library
```

CC_DATA_DIR default flipped from `$HOME/.claude-code-ui` to
`$CC_CWD_ROOT/_data`. New env var `CC_MEMORY_FILE` for explicit
override.

Commit: `16a0036`.

### 10. Multi-client = peers, not master/slave

The browser tab, the Telegram bot, and `cc-talk.py` are all peers of
the bridge. Whoever sends `{type:"send", id:<sid>, ...}` wins for
that turn. Every WS-connected client receives every broadcast — so a
turn initiated from the CLI streams live in any open browser tab.

Demo: ran `cc-talk.py history d6c73798` to dump the full message log
of a chat, then `cc-talk.py send d6c73798 "any telugu movies
releasing in may?"` to send into the same Worker without opening a
browser. Got the streaming reply on stdout, with the new memory
attribution working organically:

> *"Based on what I've got saved from earlier today, here's the May
> 2026 Telugu movie lineup…"*

Commit: `c3b1ecf`. Tool: `tools/cc-talk.py`.

## Operational fingerprints

A few things to grep for if something feels off:

- **401 from spawned claude**: `pgrep -f cc-server.py` and inspect
  its env (`ps eww`); look for `CLAUDE_CODE_OAUTH_TOKEN` or
  `ANTHROPIC_BASE_URL` leaking through. Re-launch from `env -i`.
- **Save never completes**: tail `~/claude-ui/bridge.log` for
  `silent save inject` (start) and look for the next `result` event.
  If the inject never fires → `w.busy` was already true. If it
  fires but `save_done` never broadcasts → reader is stuck (rare).
- **Index entry duplication**: search `long-term-memory.md` for
  `**Session**: <sid>` — there should be exactly one per saved
  session. Server logs `refresh: removed prior index entry for
  sid=…` when it actually removes a stale entry.
- **Save button stuck busy**: open DevTools console, eval
  `Array.from(savingChats)` to see which sids are believed to be
  saving. If a stuck sid is in there with no live save, the bridge
  crashed mid-save without firing `save_done`/`save_error`.

## Token leak — how it almost went wrong

I committed `LOCAL_NOTES.md` (which contains the literal Telegram
bot token) to a fresh `feat/long-term-memory` branch off `main`,
which didn't yet have the gitignore patterns from `feat/artifacts`
that exclude it. Force-pushed the corrected commit immediately, told
the user to revoke the token at `@BotFather`, and added the gitignore
patterns to this branch.

Lesson: **before `git add -A` on a new branch off main, run
`git status` and confirm nothing sensitive is staged.** Better:
include the gitignore-broadening commit (the one that adds
`LOCAL_NOTES.md`, `JOURNAL-*.md`, etc.) as the first commit on every
new branch.

## What I'd change next time

- **`Worker.silent_meta` could be a tagged enum**, not a free-form
  dict. Right now any consumer has to know the dict's keys.
- **The save prompt is ~70 lines**; could be tighter still. Most of
  the speedup came from the parallel-Write demand and the index-
  pre-create. The structural section list might be inferable.
- **`refreshSaveButton()` is called from ~8 places**; should be a
  reactive observer on `savingChats` / `sessions` rather than
  manual sprinkles.
- **`cc-talk.py` doesn't auto-resolve the active session**;
  `cc-talk.py send "your text"` could default to whatever
  `index.json` says is active.
- **Memory file is plain markdown** — fine for now, but for fuzzy
  trigger matching at scale, a lightweight embeddings index (just
  the trigger lines) would beat regex/grep.
